# 드래곤 빌리지 3 카드교환소 마스터 청사진 (MASTER_BLUEPRINT.md)

본 교환소 플랫폼 프로젝트의 전체 아키텍처 및 주요 기능 구성표입니다.

---

## 1. 프로젝트 개요
*   **목적**: 드래곤 빌리지 3 게임 유저들이 소장 중인 스티커 카드 정보(줄 수 있는 카드 / 받고 싶은 카드)를 등록하여 상호 간에 실시간 1:1 채팅 또는 외부 오픈채팅(연락하기)을 통해 신속하게 교환 매칭을 완료할 수 있도록 지원하는 웹 사이트입니다.
*   **기술 스택**: React, Supabase (클라이언트 및 실시간 DB 구독), LocalStorage (Mock 모드 폴백 지원), Vite, Lucide-React.

---

```
│   │   ├── auth/                 # 유저 인증 및 로그인, 내 정보(Drawer)
│   │   ├── basket/               # 유저의 스티커 수집 바구니 관리
│   │   ├── board/                # 자유/알 코드 독립 게시판 컴포넌트 및 서비스

│   ├── utils/                    # XSS 방지 및 슬래시 디코딩 등 보안/공통 유틸

│   └── chatService.js            # 실시간 채팅 메시지 API 통신 모듈
```

## 3. 핵심 기능 설명

### 🟢 [1] 스티커 도감 및 매칭 바구니
*   카테고리별(드래곤 알, 일반 카드 등) 도감을 클릭하여 내가 줄 수 있는 카드(Haves)와 받고 싶은 카드(Wants)를 등록합니다.
*   이 정보를 바탕으로 타인의 교환글과 **100% 매칭(양방향)** 또는 **부분 매칭(단방향)** 상태를 자동 계산해 주는 '매칭 엔진'이 상시 작동합니다.

### 💬 [2] 1:1 실시간 채팅 및 알림 배지
*   Supabase의 실시간 채널 구독(Realtime) 기능을 통해 유저가 메시지를 보낼 때 실시간 동기화되어 창에 뜹니다.
*   **프록시 바이패스 및 즉시 렌더링**: 상태성 소켓 연결의 컨텍스트 꼬임을 방지하기 위해 실시간 채널(`RealtimeChannel`) 구독은 프록시가 아닌 원본 `rawSupabase` 클라이언트를 직접 사용하여 구독 수신율을 100% 보장하며, 자신이 보낸 메시지는 네트워크 핑퐁 딜레이 없이 화면에 즉각 말풍선이 렌더링되도록 React 상태 즉시 주입(낙관적 갱신)을 지원합니다.
*   **안 읽은 알림 배지**: 채팅 플로팅 창은 물론, 상단 헤더의 유저 프로필 옆에도 실시간 안 읽은 개수가 표시됩니다. 백그라운드 탭에서도 구독이 중단되지 않고 뱃지 카운트가 정상 누적됩니다.

### 📝 [3] 독립 자유게시판 (Board)
*   교환 피드와 별도로 자유 소통 및 알 코드를 공유할 수 있는 로그인 회원용 독립 게시판입니다.
*   자신이 쓴 게시글의 **수정/삭제** 및 게시글 단위의 **댓글 CRUD(쓰기, 조회, 삭제)** 기능이 유기적으로 연결되어 작동합니다.

### 🏁 [4] 거래 완료 처리
*   거래가 성사된 교환글은 글쓴이가 `[완료]` 버튼을 눌러 상태를 `거래 완료`로 마킹할 수 있으며, 이 글은 opacity(흐리게) 처리되어 다른 유저들에게 직관적으로 노출됩니다.

## 4. 보안 및 유지관리
*   **보안 및 DB RLS (해킹 방지 및 관리자 우회)**: 데이터베이스 전체 테이블(`users`, `posts`, `post_comments`, `board_posts`, `board_comments`, `chat_rooms`, `chat_messages`)의 RLS 정책을 활성화했습니다. 프론트엔드(`supabaseClient.js`)에 ES6 Proxy를 주입하여 매 쿼리마다 로그인 정보(닉네임, 패스워드)를 암호화된 HTTP 커스텀 헤더에 동적 주입하고, DB 단의 `validate_request_user` 검증 함수를 통해 본인 데이터만 수정/삭제할 수 있도록 무단 조작 및 도청 해킹을 100% 차단합니다. 또한, 관리자(`'간장'`)의 정당한 관리 권한 행사를 보장하기 위해 닉네임이 `'간장'` 이고 비밀번호가 올바른 경우 소유권 체크 없이 삭제/수정이 가능하도록 RLS 바이패스 정책을 구현했습니다. 특히 대화방 개설 및 메시지 전송 시 발생하는 RLS 삽입 오류를 해결하기 위해 `chat_messages` 테이블의 INSERT 정책을 **"발신자 명의 사칭 방지(`validate_request_user(sender)`)"**로, `chat_rooms` 테이블의 INSERT 정책을 **"대화방 개설 신청자(구매자) 검증(`validate_request_user(buyer_nickname)`)"**으로 논리 구조를 대폭 단순화하여 전송 오류 및 대화방 생성 중단 문제를 원천적으로 해소했습니다.
*   **서버 장애 및 한도 초과 이메일 경고 시스템**: 배포(Production) 환경에서 DB 통신 에러(무료 티어 한도 초과, 네트워크 다운 등)가 발생하면, 조용히 로컬스토리지를 쓰는 눈속임 폴백을 중단하고 사용자에게 에러를 알림과 동시에 관리자 메일(`wnsdudvhkr@gmail.com`)로 장애 상세 내용이 실시간 자동 발송(Web3Forms API 연동)되도록 구축했습니다. (무분별한 발송 방지를 위해 12시간 자동 디바운싱 적용)
*   **데이터베이스 이관 및 복구**: Egress 할당량 초과로 차단된 이전 서울 DB로부터 사용자가 직접 제공한 JSON 백업 데이터(`users.json`, `posts.json`, `chat_message.json`)를 통해 **497명의 유저 정보, 403개의 교환 글, 730개의 채팅방, 3,276개의 실시간 메시지**를 신규 시드니 DB(`aws-1-ap-southeast-2`)로 100% 완전하게 복구 및 이전 완료하였습니다. 대용량 데이터를 고속 전송하기 위해 1,000개 단위 청크 벌크 `INSERT` 방식을 도입하여 단 6초 만에 무결성 이전 처리를 완성했습니다. 또한, 사고 위험이 높았던 관리자 페이지 내의 로컬스토리지 전송 버튼을 삭제하여 추가 안전장치를 강화했습니다.
*   **Egress 트래픽 최적화 (월 5GB 무료 한도 방어)**: 잦은 페이지 갱신에 따른 DB 호출 과다 및 비용(Egress) 누수를 줄이기 위해 `fetchPosts` API에 **60초 시간 기반 로컬 캐싱 시스템**을 이식하였습니다. 또한, 채팅 목록 로딩 시 전체 메시지를 1,000건씩 루프로 긁어오던 비효율을 제거하고 `chat_rooms` 테이블 단독 조회로 전면 최적화하여 DB 사용량을 획기적으로 낮췄습니다. 추가적으로, 사용자가 브라우저 탭을 보지 않거나 백그라운드로 전환했을 때(`document.visibilityState === 'hidden'`), 무거운 Presence 접속자 추적 채널은 자동 해제(unsubscribe)하여 핑퐁 패킷 낭비를 원천 차단하고 월 5GB 대역폭을 영구히 버틸 수 있도록 설계했으며, 실시간 알림음 및 뱃지 수신 채널은 백그라운드에서도 연결을 유지하여 실시간 알림 신뢰성을 보장했습니다. 채팅방 목록 리프레시 폴링 주기 역시 45초에서 **90초**로 늘려 트래픽을 아꼈습니다.
*   Supabase 접속이 차단되거나 오프라인/개발 환경일 때는 브라우저의 `localStorage`를 사용하는 모사 데이터베이스(`mockDB`)가 완전하게 자동 폴백되어 정상 구동을 보장합니다.

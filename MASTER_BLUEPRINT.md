# 드래곤 빌리지 3 카드교환소 마스터 청사진 (MASTER_BLUEPRINT.md)

본 교환소 플랫폼 프로젝트의 전체 아키텍처 및 주요 기능 구성표입니다.

---

## 1. 프로젝트 개요
*   **목적**: 드래곤 빌리지 3 게임 유저들이 소장 중인 스티커 카드 정보(줄 수 있는 카드 / 받고 싶은 카드)를 등록하여 상호 간에 실시간 1:1 채팅 또는 외부 오픈채팅(연락하기)을 통해 신속하게 교환 매칭을 완료할 수 있도록 지원하는 웹 사이트입니다.
*   **기술 스택**: React, Supabase (클라이언트 및 실시간 DB 구독), LocalStorage (Mock 모드 폴백 지원), Vite, Lucide-React, Vercel.

## 3. 핵심 기능 설명

### 🟢 [1] 스티커 도감 및 매칭 바구니
*   카테고리별(드래곤 알, 일반 카드 등) 도감을 클릭하여 내가 줄 수 있는 카드(Haves)와 받고 싶은 카드(Wants)를 등록합니다.
*   이 정보를 바탕으로 타인의 교환글과 **100% 매칭(양방향)** 또는 **부분 매칭(단방향)** 상태를 자동 계산해 주는 '매칭 엔진'이 상시 작동합니다.

### 💬 [2] 1:1 실시간 채팅 및 알림 배지
*   Supabase의 실시간 채널 구독(Realtime) 기능을 통해 유저가 메시지를 보낼 때 실시간 동기화되어 창에 뜹니다.
*   **인증 연동 실시간 채널 및 통합 메시지 렌더링**: DB RLS(Row Level Security)가 활성화되어 있어 실시간 채널 구독(Realtime)도 로그인 유저의 인증 정보가 있어야 변경 이벤트를 수신할 수 있습니다. 이를 해결하기 위해 `getActiveSupabaseClient()`로 획득한 인증된 클라이언트를 사용하여 실시간 채널을 구독합니다. 또한, 개별 대화방 리스너와 글로벌 알림 리스너의 다중 구독에 의한 소켓 이벤트 누락 및 채널 충돌을 방지하기 위해 글로벌 알림 리스너(`subscribeAllMyMessages`) 하나만 활성화하고, 현재 켜져 있는 활성 대화방에 매칭되는 수신 메시지를 즉시 화면 상태에 주입하는 단일 리스너 통합 구조를 적용했습니다. 한글 닉네임 유저의 채널명 충돌 방지를 위해 Base64 고유 채널명 생성 방식을 사용합니다.
*   **안 읽은 알림 배지**: 채팅 플로팅 창은 물론, 상단 헤더의 유저 프로필 옆에도 실시간 안 읽은 개수가 표시됩니다. 백그라운드 탭에서도 구독이 중단되지 않고 뱃지 카운트가 정상 누적됩니다.

### 📝 [3] 독립 자유게시판 (Board)
*   교환 피드와 별도로 자유 소통 및 알 코드를 공유할 수 있는 로그인 회원용 독립 게시판입니다.
*   자신이 쓴 게시글의 **수정/삭제** 및 게시글 단위의 **댓글 CRUD(쓰기, 조회, 삭제)** 기능이 유기적으로 연결되어 작동합니다.

### 🏁 [4] 거래 완료 처리
*   거래가 성사된 교환글은 글쓴이가 `[완료]` 버튼을 눌러 상태를 `거래 완료`로 마킹할 수 있으며, 이 글은 opacity(흐리게) 처리되어 다른 유저들에게 직관적으로 노출됩니다.

## 4. 보안 및 유지관리
*   **보안 및 DB RLS (해킹 방지 및 관리자 우회)**: 데이터베이스 전체 테이블(`users`, `posts`, `post_comments`, `board_posts`, `board_comments`, `chat_rooms`, `chat_messages`)의 RLS 정책을 활성화했습니다. 프론트엔드(`supabaseClient.js`)에 ES6 Proxy를 주입하여 매 쿼리마다 로그인 정보(닉네임, 패스워드)를 암호화된 HTTP 커스텀 헤더에 동적 주입하고, DB 단의 `validate_request_user` 검증 함수를 통해 본인 데이터만 수정/삭제할 수 있도록 무단 조작 및 도청 해킹을 100% 차단합니다. 또한, 관리자(`'간장'`)의 정당한 관리 권한 행사를 보장하기 위해 닉네임이 `'간장'` 이고 비밀번호가 올바른 경우 소유권 체크 없이 삭제/수정이 가능하도록 RLS 바이패스 정책을 구현했습니다. 특히 대화방 개설 및 메시지 전송 시 발생하는 RLS 삽입 오류를 해결하기 위해 `chat_messages` 테이블의 INSERT 정책을 **"발신자 명의 사칭 방지(`validate_request_user(sender)`)"**로, `chat_rooms` 테이블의 INSERT 정책을 **"대화방 개설 신청자(구매자) 검증(`validate_request_user(buyer_nickname)`)"**으로 논리 구조를 대폭 단순화하여 전송 오류 및 대화방 생성 중단 문제를 원천적으로 해소했습니다.
*   **서버 장애 및 한도 초과 이메일 경고 시스템**: 배포(Production) 환경에서 DB 통신 에러(무료 티어 한도 초과, 네트워크 다운 등)가 발생하면, 조용히 로컬스토리지를 쓰는 눈속임 폴백을 중단하고 사용자에게 에러를 알림과 동시에 관리자 메일(`wnsdudvhkr@gmail.com`)로 장애 상세 내용이 실시간 자동 발송(Web3Forms API 연동)되도록 구축했습니다. (무분별한 발송 방지를 위해 12시간 자동 디바운싱 적용)
*   **데이터베이스 이관 및 복구**: Egress 할당량 초과로 차단된 이전 서울 DB로부터 사용자가 직접 제공한 JSON 백업 데이터(`users.json`, `posts.json`, `chat_message.json`)를 통해 **497명의 유저 정보, 403개의 교환 글, 730개의 채팅방, 3,276개의 실시간 메시지**를 신규 시드니 DB(`aws-1-ap-southeast-2`)로 100% 완전하게 복구 및 이전 완료하였습니다. 대용량 데이터를 고속 전송하기 위해 1,000개 단위 청크 벌크 `INSERT` 방식을 도입하여 단 6초 만에 무결성 이전 처리를 완성했습니다. 또한, 사고 위험이 높았던 관리자 페이지 내의 로컬스토리지 전송 버튼을 삭제하여 추가 안전장치를 강화했습니다.
*   **Egress 트래픽 최적화 (월 5GB 무료 한도 방어)**: 잦은 페이지 갱신에 따른 DB 호출 과다 및 비용(Egress) 누수를 줄이기 위해 `fetchPosts` API에 **60초 시간 기반 로컬 캐싱 시스템**을 이식하였습니다. 또한, 채팅 목록 로딩 시 전체 메시지를 1,000건씩 루프로 긁어오던 비효율을 제거하고 `chat_rooms` 테이블 단독 조회로 전면 최적화하여 DB 사용량을 획기적으로 낮췄습니다. 추가적으로, 사용자가 브라우저 탭을 보지 않거나 백그라운드로 전환했을 때(`document.visibilityState === 'hidden'`), 무거운 Presence 접속자 추적 채널은 자동 해제(unsubscribe)하여 핑퐁 패킷 낭비를 원천 차단하고 월 5GB 대역폭을 영구히 버틸 수 있도록 설계했으며, 실시간 알림음 및 뱃지 수신 채널은 백그라운드에서도 연결을 유지하여 실시간 알림 신뢰성을 보장했습니다. 채팅방 목록 리프레시 폴링 주기 역시 45초에서 **90초**로 늘려 트래픽을 아꼈습니다. 더불어, **관리자 대시보드 통계 조회(Dashboard Stats)** 시 데이터가 1000개에서 멈추는 한계를 극복하고 트래픽 낭비를 막기 위해, 전체 포스트/메시지/방문 수는 `head: true` 카운트 쿼리를 사용하고, 전체 고유 방문자 수는 DB 단에서 직접 연산하는 RPC(`get_unique_visitors_count`)를 생성해 단 한 번에 연산하도록 개편하였으며, 일별 상세 추이는 최근 7일치 원시 데이터만 선별 쿼리하도록 제한하여 획기적으로 Egress를 방어했습니다.
*   Supabase 접속이 차단되거나 오프라인/개발 환경일 때는 브라우저의 `localStorage`를 사용하는 모사 데이터베이스(`mockDB`)가 완전하게 자동 폴백되어 정상 구동을 보장합니다.

## 5. 구글 애드센스 승인 및 크롤링(SEO) 최적화
*   **고품질 정보성 정적 HTML 페이지 구축**: SPA(Single Page Application)의 한계와 구글 애드센스 거절(가치 없는 콘텐츠, 빈 화면 광고) 문제를 극복하기 위해 `public/` 디렉토리에 8종의 정적 HTML 문서를 구현했습니다. 특히, 구글 봇이 분석할 수 있는 실질적인 정보 가치를 극대화하기 위해 유저가 제공한 데이터와 판매 환급가를 정밀 연산한 **'드빌3 젬 강화 효율 분석 가이드(tips-gem-reinforcement.html)'**를 정적 리포트로 완벽히 포매팅하였으며, **수집 효율 공략집(tips-sticker-collection.html, 1,200자 이상)**, **안전 거래 행동 수칙(tips-safe-trading.html, 1,200자 이상)**, **자동 매칭 알고리즘 가이드(tips-matching-guide.html, 1,200자 이상)** 등의 전문 팁 페이지의 텍스트 볼륨을 대규모 보강했습니다.
*   **크롤링 연동성 보강**: 신설된 모든 정적 문서를 반영해 `public/sitemap.xml` 및 `public/robots.txt`를 최신화하여 검색 엔진이 누락 없이 인덱싱하도록 설계했습니다.
*   **푸터 및 메인 네비게이션 개선**: 기존 React 앱 푸터 영역의 팝업 모달을 100% 앵커 태그(`<a>` target="_blank") 기반 정적 링크로 교체하였고, 메인 화면 상단 영역에 다크 모드 맞춤형 **'공략 & 젬 강화 효율표 퀵 바로가기 배너'**를 추가 연동하여 유저들의 자연스러운 정보 탐색을 유도하고 사이트 체류시간(Dwell Time) 지표를 획기적으로 상승하도록 유도했습니다.

## 6. 화이트모드(라이트 모드) 및 버튼 스타일 최종 튜닝
*   **가독성 극대화 및 고대 대비 보장**: 라이트 모드일 때 투명한 유리 디자인 때문에 글씨가 묻히는 것을 막기 위해 `.glass-card` 요소의 투명을 제거하고 완전 불투명(#ffffff)으로 고정했습니다. 흐릿하게 보였던 텍스트(연한 노랑, 민트, 회색)를 짙은 포레스트 그린 및 깊은 슬레이트 회색으로 변경했습니다.
*   **스티커 태그 가독성 100% 확보**: 라이트 모드 시 `줄 수 있는 스티커` 태그는 **짙은 청록색 텍스트(`#047857`) + 연한 민트 배경(`#ecfdf5`)**으로, `받고 싶은 스티커` 태그는 **짙은 다홍색 텍스트(`#b91c1c`) + 연한 핑크 배경(`#fff5f5`)**으로 개별 조정하여 가독성을 극대화했습니다.
*   **버튼 디자인의 태그 스타일 일원화**: `🔐 로그인 / 가입`, `전체등록글 보기`, `+글쓰기`, `1:1 채팅` 버튼들의 디자인을 **스티커 태그와 통일감 있게 "연한 배경 + 뚜렷한 테두리 + 짙은 글자색" 구조의 버블(태그) 버튼 형태로 일괄 동기화**했습니다. 다크 모드는 고유 브랜드 컬러인 형광 연두색(`#85c300`)을 유지하고, 라이트 모드는 밝고 산뜻한 중간 숲색 올리브 톤(`#689f38`)을 사용하여 디자인 아이덴티티를 확립했습니다.
*   **플로팅 채팅 위젯 말풍선 복원**: 채팅방 FAB 플로팅 버튼의 말풍선 아이콘 및 알림 토스트 아이콘의 글자색이 숲색 배경에 묻혀 보이지 않던 문제를 개선하기 위해 흰색(`#ffffff`)으로 완벽히 복원했습니다.
*   **Vercel 자동 배포 트리거링**: 모든 개발 변경 사항을 로컬 `feature/adsense-renewal` 브랜치에 단일 커밋 및 원격 저장소에 푸시한 뒤, 프로덕션 배포 브랜치인 `master` 브랜치로 소스코드를 성공적으로 병합(Merge) 및 푸시하여 실서버 웹버전이 최신 상태로 즉시 배포되도록 구조를 정돈했습니다.
